Introduction

Standardized test results in New York State are reported as the percentage of students scoring proficient on a given assessment, broken down by school, grade-level exam, and demographic subgroup. This competition tasked us with predicting that proficiency percentage from school-level characteristics, demographic composition, and assessment metadata. On its surface this is a routine regression problem, but several features of the dataset make this a more complex challenge. First, the target is bounded and bimodal. Rather than a smooth distribution, proficiency rates pile up at the extremes: a non-trivial share of rows report exactly 0% or exactly 100% proficient. A standard regression model trained on squared error is pulled toward the conditional mean and systematically struggles to commit to these endpoints, smoothing its predictions toward the middle where it is penalized least. Second, the data contained features that had extremely high cardinality, for example the school columns contained over 4000 unique categories. Since these features carry high predictive signal dropping them was out of the question. In order to sufficiently solve the problem of predicting school performance, we need a model that is able to address the above concerns.

The Approach: Trees & Networks

In the search of finding an appropriate model, we often start with the simplest first and then increase the complexity. Starting with a simpler linear Regression methods like Ridge and Elastic Net I found this was perhaps not the correct model for this problem. Moving to a more statistically appropriate approach in Beta Regression, I found this was also not enough juice for this data. Ultimately we needed something that could handle non-linearity and high cardinal categorical features. Two approaches showed high promise on paper and were ultimately included in the final modeling pipeline. The first being Light Gradient Boosting Model (LGBM). This model uses a gradient boosting trees approach with a couple of main differences that make this model especially good for this data. Firstly, unlike its predecessor XG-Boost, it grows leaf-wise instead of level wise allowing it to be a more flexible model but more prone to over fitting. The second biggest advantage that this model has is its native categorical handling. Light-GBM splits a categorical feature by first sorting its categories by their accumulated gradient statistic, \(\dfrac{\sum_{i \in c} g_i}{\sum_{i \in c} h_i}\), and then applying a result from Fisher (1958). Once sorted, the categories are treated as if numeric and scanned for the threshold that maximizes split gain. Fisher’s result guarantees this is exact: the optimal partition is always consecutive in the sorted order, so a high-ranked and a low-ranked category are never grouped together while a middle-ranked one is split off. This reduces an otherwise exponential search over category subsets to a single linear scan. LGBM’s internal handling proved to be very strong, and on these data set performed better than target encoding these features. This might be expected with the somewhat complex methodology that is being used under the hood. If we take a step back and take a look at how LGBM is handling the high cardinality, we can see that it is essentially computing some number for each category based on loss reduction as representation for that categorical value to solve the overall problem. This leads us naturally to the next idea, which ends up being profoundly similar when using the above framing; Embedding. Embedding, like LGBM’s approach to categorical values, does a numeric representation with a key difference. Instead of representing with a scalar value, we use a vector that is learned and refined through gradient descent. As we’ll see shortly this method learns vectors such that categories that are similar will point in the same direction, and those that are very different will point in orthogonal directions in the embedding space. To apply this methodology I will implement it inside a neural network. The embedding approach proves to be very useful for this task for several reasons including predictive performance, as well as making novel and interesting discoveries about the underlying data in a way that is not typical of non-interpretable neural networks.

Light Gradient Boosting Model

The beauty of models like LGBM is that are extremely quick to implement, and despite all the complexity under the hood it essentially comes down to import machine machine.learn(X_train,Y_train). However, something that is non-trivial is the tuning of these models. Especially with a model like LGBM, there are several hyperparameters that are mandatory to tune because of its extreme flexibility. One of the best ways to do this on any dataset of moderate size is to use a Bayesian Optimization approach to tune the hyperparamters, implemented through Optuna. The particular Optuna loop that I settled on was a 6 fold cross-validation within, with Optuna optimizing for the parameters that perform the best the average across the CV fold. The idea being that Optuna will pick the hyperparameters that generalize the best. This model ultimately ended with an MSE of 82 on the internal test set and gave the following results.

Neural Network with Embedding

With this next model, I hoped to go deeper into the data and capture interactions and relationships that LGBM and other tree-based models were missing. After setting up the data to use embeddings and building out the architecture, the model immediately showed promise, landing within striking range of the tree-based models. The common sentiment is that tree-based models tend to beat neural networks on tabular data, so this was a welcome surprise. The advantage (and disadvantage) of a neural network is that it is fully customizable: you design the model from the ground up, including how it trains. This is where the bulk of the effort went, finding the optimal design choices.

Architectural Choices

The first neural architecture, EmbedNet, established the core design that all later versions retained: separate learned embeddings for categorical features (school, district, district type, assessment, and subgroup), each processed through its own small feedforward “tower” before concatenation with transformed continuous features and input into a deep merge network. This design allows each feature group to learn intermediate representations in isolation before interacting jointly in the final layers, where higher-order relationships between features are learned. Embedding dropout was applied to high-cardinality categorical variables (school, district, and exam) to reduce overfitting, as these features are most prone to memorizing individual categories. In addition, two bilinear interaction layers explicitly modeled pairwise relationships between schools and exams, and between subgroups and exams, encoding the intuition that performance depends on specific feature interactions rather than independent effects alone. By doing this we can avoid hoping that the network will learn this interaction on its own, and instead give it directly. The merge network itself was deep and wide, consisting of five fully connected blocks that progressively reduced dimensionality from 512 units to a single output, with a final sigmoid activation scaled to match the target range [0, 100]. This architecture remained stable through initial experimentation, but further improvements via hyperparameter tuning and regularization alone proved insufficient. This motivated a reassessment of the model’s depth. In particular, the combination of a depth and the saturating sigmoid output suggested that signal propagation through the network may degrade with depth, limiting the effectiveness of early layers. To investigate this, we analyze activation variance across layers during forward passes to see how signal strength changes with depth. This gives a direct way to check whether information is preserved or gradually lost as it moves through the network.

import torch
import torch.nn as nn

def activation_variance_depth_trace(
    model,
    loader,
    device="mps",
    n_batches=50,
    title="Activation Variance Across Network Depth"
):

    model = model.to(device)
    model.eval()

    activations = {}
    hooks = []
    layer_order = []

    
    # identify all linear layers in forward order
    
    for name, module in model.named_modules():
        if isinstance(module, torch.nn.Linear):
            layer_order.append(name)
            activations[name] = []

    
    # hook function
    
    def make_hook(name):
        def hook(module, inp, out):
            out = out.detach()

            # variance per neuron, then average across neurons
            var = out.var(dim=0).mean().item()

            activations[name].append(var)

        return hook

    # register hooks
    for name, module in model.named_modules():
        if name in activations:
            hooks.append(module.register_forward_hook(make_hook(name)))

    
    # run forward passes
    
    with torch.no_grad():
        for i, batch in enumerate(loader):
            if i >= n_batches:
                break

            (
                school_ids,
                district_ids,
                distype_ids,
                exam_ids,
                subgroup_ids,
                cont_x,
                y
            ) = [b.to(device) for b in batch]

            _ = model(
                school_ids,
                district_ids,
                distype_ids,
                exam_ids,
                subgroup_ids,
                cont_x
            )

    # remove hooks
    for h in hooks:
        h.remove()

    
    # aggregate results
    
    layer_means = []
    layer_stds = []

    for name in layer_order:
        vals = np.array(activations[name])
        layer_means.append(vals.mean())
        layer_stds.append(vals.std())

    
    # plot
    
    x = np.arange(len(layer_order))

    fig, ax = plt.subplots(figsize=(9, 5))

    ax.plot(x, layer_means, marker="o", label="Mean Activation Variance")

    ax.fill_between(
        x,
        np.array(layer_means) - np.array(layer_stds),
        np.array(layer_means) + np.array(layer_stds),
        alpha=0.2
    )

    ax.set_xticks(x)
    ax.set_xticklabels(layer_order, rotation=45, ha="right")

    ax.set_xlabel("Layer (Depth Order)")
    ax.set_ylabel("Activation Variance")
    ax.set_title(title)

    ax.grid(True, alpha=0.2)
    ax.legend()

    plt.tight_layout()
    plt.show()

    return {
        "layer_order": layer_order,
        "means": layer_means,
        "stds": layer_stds,
        "raw": activations
    }

We can see that the activation variance of some of the layers is close to zero, suggesting that they are not reacting to input differences, and the representations are becoming uniform, this is especially true for the earlier layers. This motivates some more architectural changes. The first change was to remove the towers all together, and slowly add them back only when there was an improvement in loss. This showed some surprising results. Removing the towers altogether resulted a in a marginally better or similar result. We can make the assumption based on both the increased accuracy after the removal of the towers and gradient flow plot that the earlier layer were not getting enough signal. With this conclusion i decided to update the size of the embeddings moving the school embedding from size 16 to 50 and eventually to 64. This resulted in the single biggest reduction of loss in the iterations of the model. After updating all the embedding sizes I was able to get to the mid to high 50s in MSE before revisiting the towers again. Another a few trials on tweaking the towers revealed that smaller, simpler towers ended up improving performance. We can perhaps make the conclusion that the refinement taking place in the towers is actually good, but gradient problems lead to a cap on performance with the towers. This is where the idea of skip or residual connections is extremely useful. Skip connections allow for us pass the original input alongside the network transformed output. For example using the school tower as an example we have: \(f_{s}(x) + x\) where x is the school embedding. After a few interactions we land on a final model that contained larger embeddings with smaller towers for each, a wider merged section, and skip connections on the school, exam, subgroup, and continuos towers. Interestingly enough the bilinear interaction layers perform better on the raw embeddings, than the tower transformed embeddings. Let’s take a look at the gradient flow of this model.

            school_out, district_emb, distype_emb,
            exam_out, subgroup_out,
            school_exam_out, subgroup_exam_out,
            continuous_out,
        ], dim=1)

        block1_out = self.merge_block1(merged) + self.skip_proj(merged)
        out = self.merge_block2(block1_out)
        return torch.sigmoid(out) * 100.0

    def predict(self, loader, device="cpu"):
        self.eval()
        all_preds = []
        with torch.no_grad():
            for school_ids, district_ids, distype_ids, exam_ids, subgroup_ids, cont_x, _ in loader:
                school_ids   = school_ids.to(device)
                district_ids = district_ids.to(device)
                distype_ids  = distype_ids.to(device)
                exam_ids     = exam_ids.to(device)
                subgroup_ids = subgroup_ids.to(device)
                cont_x       = cont_x.to(device)
                preds = self.forward(school_ids, district_ids, distype_ids, exam_ids, subgroup_ids, cont_x)
                all_preds.append(preds.cpu())
        return torch.cat(all_preds).numpy().flatten()

Struggles, pitfalls and Possible Improvements

The most persistent technical failure was capturing the endpoints of the distribution. Despite building specialized models aimed directly at this problem — asymmetric-loss boosters that penalized under-prediction to chase the upper spike, and the mirror version for the lower spike — and ensembling them alongside the general models, the final predictions still smoothed toward the middle of the range. The specialists shifted predictions in the right direction but never committed to exact 0 or 100 values, and the stacker, by averaging across base models, pulled even the confident extremes back toward the interior. In hindsight this is an inherent tension: a model trained to minimize squared error has no incentive to output an exact boundary value, because the conditional mean of a noisy, bounded target almost never sits precisely at the edge. No amount of ensembling resolved this, because every base model shared the same underlying disincentive.

A second, more process-oriented weakness was how I allocated my time. Early on, I invested heavily in tuning LightGBM’s hyperparameters close to exhaustion, sweeping learning rates, leaf counts, regularization, and sampling fractions in pursuit of marginal gains. The returns on this effort were small; once the model was reasonably configured, further tuning moved the validation error by fractions of a point. The lesson, which I would apply differently next time, is that breadth beats depth in the early and middle stages of a competition. Time spent exhausting one model’s hyperparameter space would have been better spent trialing genuinely different approaches, the diversity of model types contributed far more to the final ensemble than the fine-tuning of any single one. For example, I discovered using the embeddings from the neural net to replace the respective columns in the dataset, and then training tree based models on that data resulted in a much stronger tree model. This concept opened up a wide range of models since it eliminated the high cardinal variables from vectors, but was implemented to late in the game.

Another issue I faced throughout the modeling process was a persistent train–test gap. This was not typical overfitting: no amount of regularization closed it. When training error fell, test error fell in step, but the gap between them held nearly constant at around 30 MSE. What explains a gap that moves in parallel with training error rather than widening as the model fits? It could be irreducible noise concentrated in the test set. A secondary possibility is that the train–test split left some embedding categories sparse: with fewer training rows per school, the learned embeddings are noisier, and schools appearing rarely (or only) in the test split cannot be represented well. This is something that I still need to understand.

The most important lesson, however, came from a structural property of the data discovered after the fact. The target is not a set of independent observations: subgroup proficiency rates and their student counts satisfy an exact, count-weighted identity with their parent group. Recognizing this allowed a large fraction of the predictions to be solved algebraically, rather than estimated by any model. Applied as a post-competition submission, this single insight reduced the error to roughly 25 MSE, well below anything the modeling alone achieved. The gap between what the models produced and what the algebra recovered is the clearest possible evidence that the largest gains in this problem came not from model sophistication but from understanding how the data was generated. Had I investigated the relational structure of the target earlier, I would have reached this result during the competition rather than after it.

# Introduction
Standardized test results in New York State are reported as the percentage of students scoring proficient on a given assessment, broken down by school, grade-level exam, and demographic subgroup. This competition tasked us with predicting that proficiency percentage from school-level characteristics, demographic composition, and assessment metadata. On its surface this is a routine regression problem, but several features of the dataset make this a more complex challenge.
First, the target is bounded and bimodal. Rather than a smooth distribution, proficiency rates pile up at the extremes: a non-trivial share of rows report exactly 0% or exactly 100% proficient. A standard regression model trained on squared error is pulled toward the conditional mean and systematically struggles to commit to these endpoints, smoothing its predictions toward the middle where it is penalized least. Second, the data contained features that had extremely high cardinality, for example the school columns contained over 4000 unique categories. Since these features carry high predictive signal dropping them was out of the question. In order to sufficiently solve the problem of predicting school performance, we need a model that is able to address the above concerns. 

### *The Approach: Trees & Networks*
In the search of finding an appropriate model, we often start with the simplest first and then increase the complexity. Starting with a simpler linear Regression methods like Ridge and Elastic Net I found this was perhaps not the correct model for this problem. Moving to a more statistically appropriate approach in Beta Regression, I found this was also not enough juice for this data. Ultimately we needed something that could handle non-linearity and high cardinal categorical features. Two approaches showed high promise on paper and were ultimately included in the final modeling pipeline. The first being Light Gradient Boosting Model (LGBM). This model uses a gradient boosting trees approach with a couple of main differences that make this model especially good for this data. Firstly, unlike its predecessor XG-Boost, it grows leaf-wise instead of level wise allowing it to be a more flexible model but more prone to over fitting. The second biggest advantage that this model has is its native categorical handling. Light-GBM splits a categorical feature by first sorting its categories by their accumulated gradient statistic, $\dfrac{\sum_{i \in c} g_i}{\sum_{i \in c} h_i}$, and then applying a result from Fisher (1958). Once sorted, the categories are treated as if numeric and scanned for the threshold that maximizes split gain. Fisher's result guarantees this is exact: the optimal partition is always *consecutive* in the sorted order, so a high-ranked and a low-ranked category are never grouped together while a middle-ranked one is split off. This reduces an otherwise exponential search over category subsets to a single linear scan. LGBM's internal handling proved to be very strong, and on these data set performed better than target encoding these features. This might be expected with the somewhat complex methodology that is being used under the hood. If we take a step back and take a look at how LGBM is handling the high cardinality, we can see that it is essentially computing some number for each category based on loss reduction as representation for that categorical value to solve the overall problem. This leads us naturally to the next idea, which ends up being profoundly similar when using the above framing; Embedding. Embedding, like LGBM's approach to categorical values, does a numeric representation with a key difference. Instead of representing with a scalar value, we use a vector that is learned and refined through gradient descent. As we'll see shortly this method learns vectors such that categories that are similar will point in the same direction, and those that are very different will point in orthogonal directions in the embedding space. To apply this methodology I will implement it inside a neural network. The embedding approach proves to be very useful for this task for several reasons including predictive performance, as well as making novel and interesting discoveries about the underlying data in a way that is not typical of non-interpretable neural networks.

With this next model, I hoped to go deeper into the data and capture interactions and relationships that LGBM and other tree-based models were missing. After setting up the data to use embeddings and building out the architecture, the model immediately showed promise, landing within striking range of the tree-based models. The common sentiment is that tree-based models tend to beat neural networks on tabular data, so this was a welcome surprise. The advantage (and disadvantage) of a neural network is that it is fully customizable: you design the model from the ground up, including how it trains. This is where the bulk of the effort went, finding the optimal design choices.

#### *Architectural Choices*

The first neural architecture, EmbedNet, established the core design that all later versions retained: separate learned embeddings for categorical features (school, district, district type, assessment, and subgroup), each processed through its own small feedforward “tower” before concatenation with transformed continuous features and input into a deep merge network. This design allows each feature group to learn intermediate representations in isolation before interacting jointly in the final layers, where higher-order relationships between features are learned. Embedding dropout was applied to high-cardinality categorical variables (school, district, and exam) to reduce overfitting, as these features are most prone to memorizing individual categories. In addition, two bilinear interaction layers explicitly modeled pairwise relationships between schools and exams, and between subgroups and exams, encoding the intuition that performance depends on specific feature interactions rather than independent effects alone. By doing this we can avoid hoping that the network will learn this interaction on its own, and instead give it directly. The merge network itself was deep and wide, consisting of five fully connected blocks that progressively reduced dimensionality from 512 units to a single output, with a final sigmoid activation scaled to match the target range [0, 100]. This architecture remained stable through initial experimentation, but further improvements via hyperparameter tuning and regularization alone proved insufficient. This motivated a reassessment of the model’s depth. In particular, the combination of a depth and the saturating sigmoid output suggested that signal propagation through the network may degrade with depth, limiting the effectiveness of early layers. To investigate this, we analyze activation variance across layers during forward passes to see how signal strength changes with depth. This gives a direct way to check whether information is preserved or gradually lost as it moves through the network.

import torch
import torch.nn as nn
import numpy as np
import matplotlib.pyplot as plt
import copy

def activation_variance_depth_trace(
    model,
    loader,
    device="mps",
    n_batches=50,
    title="Activation Variance Across Network Depth"
):

    model = model.to(device)
    model.eval()

    activations = {}
    hooks = []
    layer_order = []

    
    # identify all linear layers in forward order
    
    for name, module in model.named_modules():
        if isinstance(module, torch.nn.Linear):
            layer_order.append(name)
            activations[name] = []

    
    # hook function
    
    def make_hook(name):
        def hook(module, inp, out):
            out = out.detach()

            # variance per neuron, then average across neurons
            var = out.var(dim=0).mean().item()

            activations[name].append(var)

        return hook

    # register hooks
    for name, module in model.named_modules():
        if name in activations:
            hooks.append(module.register_forward_hook(make_hook(name)))

    
    # run forward passes
    
    with torch.no_grad():
        for i, batch in enumerate(loader):
            if i >= n_batches:
                break

            (
                school_ids,
                district_ids,
                distype_ids,
                exam_ids,
                subgroup_ids,
                cont_x,
                y
            ) = [b.to(device) for b in batch]

            _ = model(
                school_ids,
                district_ids,
                distype_ids,
                exam_ids,
                subgroup_ids,
                cont_x
            )

    # remove hooks
    for h in hooks:
        h.remove()

    
    # aggregate results
    
    layer_means = []
    layer_stds = []

    for name in layer_order:
        vals = np.array(activations[name])
        layer_means.append(vals.mean())
        layer_stds.append(vals.std())

    
    # plot
    
    x = np.arange(len(layer_order))

    fig, ax = plt.subplots(figsize=(9, 5))

    ax.plot(x, layer_means, marker="o", label="Mean Activation Variance")

    ax.fill_between(
        x,
        np.array(layer_means) - np.array(layer_stds),
        np.array(layer_means) + np.array(layer_stds),
        alpha=0.2
    )

    ax.set_xticks(x)
    ax.set_xticklabels(layer_order, rotation=45, ha="right")

    ax.set_xlabel("Layer (Depth Order)")
    ax.set_ylabel("Activation Variance")
    ax.set_title(title)

    ax.grid(True, alpha=0.2)
    ax.legend()

    plt.tight_layout()
    plt.show()

    return {
        "layer_order": layer_order,
        "means": layer_means,
        "stds": layer_stds,
        "raw": activations
    }

We can see that the activation variance of some of the layers is close to zero, suggesting that they are not reacting to input differences, and the representations are becoming uniform, this is especially true for the earlier layers. This motivates some more architectural changes. The first change was to remove the towers all together, and slowly add them back only when there was an improvement in loss. This showed some surprising results. Removing the towers altogether resulted a in a marginally better or similar result. We can make the assumption based on both the increased accuracy after the removal of the towers and gradient flow plot that the earlier layer were not getting enough signal. With this conclusion i decided to update the size of the embeddings moving the school embedding from size 16 to 50 and eventually to 64. This resulted in the single biggest reduction of loss in the iterations of the model. After updating all the embedding sizes I was able to get to the mid to high 50s in MSE before revisiting the towers again. Another a few trials on tweaking the towers revealed that smaller, simpler towers ended up improving performance. We can perhaps make the conclusion that the refinement taking place in the towers is actually good, but gradient problems lead to a cap on performance with the towers. This is where the idea of skip or residual connections is extremely useful. Skip connections allow for us pass the original input alongside the network transformed output. For example using the school tower as an example we have: $f_{s}(x) + x$ where x is the school embedding. After a few interactions we land on a final model that contained larger embeddings with smaller towers for each, a wider merged section, and skip connections on the school, exam, subgroup, and continuos towers. class SkipEmbed_V2(nn.Module):
    def __init__(self,
                 n_schools,
                 n_districts,
                 n_distype,
                 n_exam,
                 n_subgroups,
                 n_continuous,
                 school_embed_dim=50,
                 district_embed_dim=20,
                 distype_embed_dim=8,
                 exam_embed_dim=35,
                 subgroup_embed_dim=20,
                 school_embed_dropout=0.10,
                 district_embed_dropout=0.05,
                 exam_embed_dropout=0.08,
                 cont_dropout_rate=0.35):   #from 0.15 — main change for richer features

        super().__init__()

        # ── embedding layers (unchanged)
        self.school_embed   = nn.Embedding(n_schools,   school_embed_dim)
        self.district_embed = nn.Embedding(n_districts, district_embed_dim)
        self.distype_embed  = nn.Embedding(n_distype,   distype_embed_dim)
        self.exam_embed     = nn.Embedding(n_exam,      exam_embed_dim)
        self.subgroup_embed = nn.Embedding(n_subgroups, subgroup_embed_dim)

        # ── school tower (unchanged)
        self.school_tower = nn.Sequential(
            nn.Linear(school_embed_dim, 32),
            nn.BatchNorm1d(32),
            nn.ReLU(),
            nn.Dropout(0.05),
            nn.Linear(32, school_embed_dim),
        )
        self.school_tower_skip = nn.Linear(school_embed_dim, school_embed_dim, bias=False)

        # ── exam tower (unchanged)
        self.exam_tower = nn.Sequential(
            nn.Linear(exam_embed_dim, 16),
            nn.BatchNorm1d(16),
            nn.ReLU(),
            nn.Dropout(0.05),
            nn.Linear(16, exam_embed_dim),
        )
        self.exam_tower_skip = nn.Linear(exam_embed_dim, exam_embed_dim, bias=False)

        # ── subgroup tower (unchanged)
        self.subgroup_tower = nn.Sequential(
            nn.Linear(subgroup_embed_dim, 8),
            nn.BatchNorm1d(8),
            nn.ReLU(),
            nn.Dropout(0.05),
            nn.Linear(8, subgroup_embed_dim),
        )
        self.subgroup_skip = nn.Linear(subgroup_embed_dim, subgroup_embed_dim, bias=False)

        # ── embedding dropouts (unchanged)
        self.school_embed_drop   = nn.Dropout(school_embed_dropout)
        self.district_embed_drop = nn.Dropout(district_embed_dropout)
        self.exam_embed_drop     = nn.Dropout(exam_embed_dropout)

        # ── bilinear interactions (unchanged)
        self.school_exam_bilinear   = nn.Bilinear(school_embed_dim,   exam_embed_dim, 16)
        self.subgroup_exam_bilinear = nn.Bilinear(subgroup_embed_dim, exam_embed_dim, 8)

        # ── continuous tower — widened and more regularized
        self.continuous_tower = nn.Sequential(
            nn.Linear(n_continuous, 512),           # from 128
            nn.BatchNorm1d(512),
            nn.ReLU(),
            nn.Dropout(cont_dropout_rate),          # from 0.25 (via cont_dropout_rate default)

            nn.Linear(512, 128),                    # new intermediate layer
            nn.BatchNorm1d(128),
            nn.ReLU(),
            nn.Dropout(cont_dropout_rate * 0.6),    # tapered dropout

            nn.Linear(128, 64),
            nn.BatchNorm1d(64),
            nn.ReLU(),
            nn.Dropout(0.10),

            nn.Linear(64, 32),
        )

        # ── continuous skip (unchanged)
        self.cont_skip_proj = nn.Linear(n_continuous, 32, bias=False)

        # ── merged size (unchanged formula)
        merged_size = (school_embed_dim + district_embed_dim +
                       distype_embed_dim + exam_embed_dim +
                       subgroup_embed_dim +
                       16 + 8 + 32)
        print(f"merged_size: {merged_size}")

        # ── merge block 1 (unchanged)
        self.merge_block1 = nn.Sequential(
            nn.Linear(merged_size, 1024),
            nn.BatchNorm1d(1024),
            nn.ReLU(),
            nn.Dropout(0.25),
        )
        self.skip_proj = nn.Linear(merged_size, 1024, bias=False)

        # ── merge block 2 (unchanged)
        self.merge_block2 = nn.Sequential(
            nn.Linear(1024, 512),
            nn.BatchNorm1d(512),
            nn.ReLU(),
            nn.Dropout(0.20),

            nn.Linear(512, 64),
            nn.BatchNorm1d(64),
            nn.ReLU(),

            nn.Linear(64, 1),
        )

    def forward(self, school_ids, district_ids, distype_ids, exam_ids, subgroup_ids, continuous_features):
        school_emb   = self.school_embed_drop(self.school_embed(school_ids))
        district_emb = self.district_embed_drop(self.district_embed(district_ids))
        distype_emb  = self.distype_embed(distype_ids)
        exam_emb     = self.exam_embed_drop(self.exam_embed(exam_ids))
        subgroup_emb = self.subgroup_embed(subgroup_ids)

        school_out   = self.school_tower(school_emb)   + self.school_tower_skip(school_emb)
        exam_out     = self.exam_tower(exam_emb)       + self.exam_tower_skip(exam_emb)
        subgroup_out = self.subgroup_tower(subgroup_emb) + self.subgroup_skip(subgroup_emb)

        continuous_out = self.continuous_tower(continuous_features) + \
                         self.cont_skip_proj(continuous_features)

        school_exam_out   = self.school_exam_bilinear(school_emb, exam_emb)
        subgroup_exam_out = self.subgroup_exam_bilinear(subgroup_emb, exam_emb)

        merged = torch.cat([
            school_out, district_emb, distype_emb,
            exam_out, subgroup_out,
            school_exam_out, subgroup_exam_out,
            continuous_out,
        ], dim=1)

        block1_out = self.merge_block1(merged) + self.skip_proj(merged)
        out = self.merge_block2(block1_out)
        return torch.sigmoid(out) * 100.0

    def predict(self, loader, device="cpu"):
        self.eval()
        all_preds = []
        with torch.no_grad():
            for school_ids, district_ids, distype_ids, exam_ids, subgroup_ids, cont_x, _ in loader:
                school_ids   = school_ids.to(device)
                district_ids = district_ids.to(device)
                distype_ids  = distype_ids.to(device)
                exam_ids     = exam_ids.to(device)
                subgroup_ids = subgroup_ids.to(device)
                cont_x       = cont_x.to(device)
                preds = self.forward(school_ids, district_ids, distype_ids, exam_ids, subgroup_ids, cont_x)
                all_preds.append(preds.cpu())
        return torch.cat(all_preds).numpy().flatten()

def loss_landscape_smoothness(
    model,
    loader,
    device="mps",
    sigmas=None,
    n_batches=20
):
    model = model.to(device)
    model.eval()

    if sigmas is None:
        sigmas = np.linspace(0.0, 0.02, 10)

    loss_fn = torch.nn.MSELoss()

    base_state = {k: v.detach().clone() for k, v in model.state_dict().items()}
    results = []

    for sigma in sigmas:

        losses = []

        with torch.no_grad():
            for name, param in model.named_parameters():
                if param.dtype.is_floating_point:
                    param.add_(torch.randn_like(param) * sigma)

        with torch.no_grad():
            for i, batch in enumerate(loader):
                if i >= n_batches:
                    break

                (
                    school_ids,
                    district_ids,
                    distype_ids,
                    exam_ids,
                    subgroup_ids,
                    cont_x,
                    y

                preds = model(
                    school_ids,
                    district_ids,
                    distype_ids,
                    exam_ids,
                    subgroup_ids,
                    cont_x
                )

                losses.append(loss_fn(preds, y).item())

        results.append(np.mean(losses))
        model.load_state_dict(base_state)

    return sigmas, results

#### *Struggles, pitfalls and Possible Improvements*

The most persistent technical failure was capturing the endpoints of the distribution. Despite building specialized models aimed directly at this problem — asymmetric-loss boosters that penalized under-prediction to chase the upper spike, and the mirror version for the lower spike — and ensembling them alongside the general models, the final predictions still smoothed toward the middle of the range. The specialists shifted predictions in the right direction but never committed to exact 0 or 100 values, and the stacker, by averaging across base models, pulled even the confident extremes back toward the interior. In hindsight this is an inherent tension: a model trained to minimize squared error has no incentive to output an exact boundary value, because the conditional mean of a noisy, bounded target almost never sits precisely at the edge. No amount of ensembling resolved this, because every base model shared the same underlying disincentive.

A second, more process-oriented weakness was how I allocated my time. Early on, I invested heavily in tuning LightGBM's hyperparameters close to exhaustion, sweeping learning rates, leaf counts, regularization, and sampling fractions in pursuit of marginal gains. The returns on this effort were small; once the model was reasonably configured, further tuning moved the validation error by fractions of a point. The lesson, which I would apply differently next time, is that breadth beats depth in the early and middle stages of a competition. Time spent exhausting one model's hyperparameter space would have been better spent trialing genuinely different approaches, the diversity of model types contributed far more to the final ensemble than the fine-tuning of any single one. For example, I discovered using the embeddings from the neural net to replace the respective columns in the dataset, and then training tree based models on that data resulted in a much stronger tree model. This concept opened up a wide range of models since it eliminated the high cardinal variables from vectors, but was implemented to late in the game. 

Another issue I faced throughout the modeling process was a persistent train–test gap. This was not typical overfitting: no amount of regularization closed it. When training error fell, test error fell in step, but the gap between them held nearly constant at around 30 MSE. What explains a gap that moves in parallel with training error rather than widening as the model fits? It could be irreducible noise concentrated in the test set. A secondary possibility is that the train–test split left some embedding categories sparse: with fewer training rows per school, the learned embeddings are noisier, and schools appearing rarely (or only) in the test split cannot be represented well. This is something that I still need to understand.

The most important lesson, however, came from a structural property of the data discovered after the fact. The target is not a set of independent observations: subgroup proficiency rates and their student counts satisfy an exact, count-weighted identity with their parent group. Recognizing this allowed a large fraction of the predictions to be solved algebraically, rather than estimated by any model. Applied as a post-competition submission, this single insight reduced the error to roughly 25 MSE, well below anything the modeling alone achieved. The gap between what the models produced and what the algebra recovered is the clearest possible evidence that the largest gains in this problem came not from model sophistication but from understanding how the data was generated. Had I investigated the relational structure of the target earlier, I would have reached this result during the competition rather than after it.
